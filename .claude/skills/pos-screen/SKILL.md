# pos-screen — загрузка спецификации экрана кассира POS

Цель: до того как агент напишет хоть строчку UI-кода на PySide6, он обязан получить **актуальную пару (frame в дизайне ↔ раздел в PRD)** и работать строго по ней. Никаких "по памяти", никаких выдуманных layout-ов, никакой реализации экранов вне MVP-скоупа.

## Что делать при вызове skill

Получив аргумент (номер 1..24 или ключевое слово), выполнить **все** шаги по порядку.

### Шаг 0. Проверить scope

Если запрошенный экран **не из MVP** (не входит в таблицу «MVP-скоуп» выше) — остановиться и сказать пользователю:

> «Экран N — Phase X, в MVP не входит. Подтвердите расширение скоупа, иначе работать не буду».

Продолжать только после явного подтверждения.

### Шаг 1. Открыть документ через pencil MCP (НЕ jq, НЕ Read)

`.pen` — зашифрованный формат, читать **только** через `mcp__pencil__*`-тулы (см. instructions сервера pencil). `Read`/`grep`/`jq` по `.pen` запрещены.

```
mcp__pencil__open_document(path="design/pos_cashier.pen")
mcp__pencil__batch_get(filePath="...", patterns=[{"name": "^4\\. POS"}], readDepth=3)
```

Если pencil MCP не подключён — остановиться и попросить пользователя его подключить. Не пытаться jq.

### Шаг 2. Получить screenshot frame'а — обязательно

```
mcp__pencil__get_screenshot(filePath="...", nodeId="<frame_id>")
```

Скриншот показывает **точные пропорции, отступы, цвета, размеры элементов**. Сравнивать визуал кода с этим скриншотом — основной критерий «делает точь-в-точь». Никаких «выдуманных» макетов: если в дизайне карточки 180×100 — в коде ровно 180×100, без `Expanding` если не указано `fill_container` явно.

Зафиксировать в ответе:
- размеры frame (`width`, `height`) — canvas экрана;
- иерархию children с именами (мапятся на `QWidget`/layouts);
- для каждого узла: bounds, style (цвета из `$variables`), text.
- **скриншот** — обязательно вложить в Шаг 5.

### Шаг 2b. Точное соответствие дизайну

- Размеры карточек/кнопок/иконок — **из дизайна**, не «удобные округлые». Если frame говорит `width: 180, height: 100` — `setFixedSize(180, 100)`.
- `fill_container` в дизайне = `QSizePolicy.Expanding` или соответствующий column/row stretch. Если в дизайне нет `fill_container` — **не делать Expanding**.
- Цвета — **только из `$variables`** (`mcp__pencil__get_variables`). Не выдумывать.
- Иконки — lucide. Маппить точное имя из `iconFontName` (например `layout-grid`, `utensils-crossed`). Если такой иконки нет в `pos/resources/icons.py` — **добавить SVG-строку**, не fallback на «utensils».
- Шрифты, размеры текста, паддинги, gap, cornerRadius — из дизайна.
- При расхождении PRD ↔ дизайн — **спросить пользователя**, не выбирать молча.

### Шаг 3. Подтянуть раздел PRD

Cashier-экраны: [prd-v3/cashier/C-01-SCREENS.md](../../../prd-v3/cashier/C-01-SCREENS.md). Найти секцию `## <номер>. <название>` или ближайшую по смыслу. Также проверить:
- [prd-v3/cashier/C-00-OVERVIEW.md](../../../prd-v3/cashier/C-00-OVERVIEW.md) — структура проекта, `state.py`, `QStackedWidget`-роутинг
- [prd-v3/cashier/C-02-HTTP-CLIENT.md](../../../prd-v3/cashier/C-02-HTTP-CLIENT.md) — какие эндпоинты дёргает экран, SSE-стрим в QThread
- [prd-v3/cashier/C-03-PRINTING.md](../../../prd-v3/cashier/C-03-PRINTING.md) — если экран связан с печатью
- backend-контракт для каждого упомянутого эндпоинта в [prd-v3/backend/](../../../prd-v3/backend/) (B-01..B-06) и сквозной [prd-v3/01-API-CONTRACT.md](../../../prd-v3/01-API-CONTRACT.md)

Если PRD-раздела для экрана нет — **остановиться** и сказать пользователю: «PRD для экрана N отсутствует, нужно дописать `prd-v3/cashier/C-01-SCREENS.md` перед UI-работой».

### Шаг 4. Сверить с уже существующим кодом

```bash
ls apps/pos/pos 2>/dev/null
ls apps/pos/pos/screens 2>/dev/null
ls apps/pos/pos/widgets 2>/dev/null
grep -rni "<keyword>" apps/pos/pos 2>/dev/null
```

Если экран частично реализован — отметить, какие файлы уже существуют и что в них.

### Шаг 5. Выдать сводку перед кодингом

Сводка обязательна и должна содержать ровно эти секции:

```
## Экран: <номер>. <название>
- Класс: <PinLoginScreen | TablesScreen | ...>
- Файл: apps/pos/pos/screens/<snake>.py (новый | существующий)
- Frame в .pen: размер WxH, структура (дерево имён)
- PRD-раздел: путь:строка, ключевые поведения (5–10 буллетов)
- API-эндпоинты: METHOD /path → из B-XX, заголовки (PIN <token>, Idempotency-Key)
- SSE-события, на которые подписан экран: table.updated / order.* / print_job.updated
- Печать: да/нет (если да — какой шаблон из C-03)
- Существующий код: список файлов или "пусто"
- План реализации: 3–7 пунктов, привязанные к компонентам из frame и сигналам PySide6
```

Только после этой сводки разрешено писать/менять код.

В коммите явно указать имя frame'а: `feat(cashier): implement screen "1. PIN Login"`.

## Адаптивность (обязательно для каждого экрана)

Все экраны кассира должны быть адаптивными — корректно работать на любом разумном размере окна и количестве данных:

- **Любые сетки/списки** (столы, активные заказы, меню) — кол-во колонок вычисляется по ширине окна (`width // (card_w + gap)`), пересчёт в `resizeEvent`. Карточки фиксированной ширины (`setFixedSize`/`setFixedWidth`); сетка даёт `Qt.AlignTop | Qt.AlignLeft`.
- **Пределы:** мин 2 колонки, макс 8 (для карты зала); 1 колонка (для списка заказов) на узком окне.
- Длинные списки заворачиваем в `QScrollArea(setWidgetResizable=True)`.
- Не использовать абсолютные координаты (`move(x, y)`) для контента. Только layouts.
- Min window size — поддерживать **1024×640** (минимальный POS-моноблок), preferred 1366×768 или 1920×1080.

## Не ломать работающее без согласования

Полное правило — в [CLAUDE.md](../../../CLAUDE.md#правило-не-ломать-рабочее-без-согласования). Кратко:

- Делай **точечное** изменение для своей задачи. Не правь «заодно» что-то соседнее.
- Если задача про экран X, **запрещено** трогать `main.py`, `state.py`, `QApplication`-уровень, `pos/widgets/*` (общие компоненты типа `Sidebar`), палитру или глобальный stylesheet — без явного предупреждения и одобрения.
- Если по ходу видишь, что **необходима** правка вне границ задачи — **остановись, опиши проблему** и попроси разрешения, не делай молча.
- После любой фичи / правки стиля — обязательно прогон полного `pytest` (POS + backend), и подтверждение «всё зелёное» в финальном ответе.
- Если упал ранее работавший тест — это **блокер**: либо немедленно фиксишь, либо откатываешь свою правку. Не двигаешься дальше.

Особенно про стили/фоны/палитру:
- `QWidget`-подкласс с class-селектор bg → нужен `setAttribute(Qt.WA_StyledBackground, True)`. Если правишь это для одного экрана — **проверь и остальные** через тест `tests/test_screens_styled_background.py`.
- Никогда не делай `app.setStyle("Fusion")` / `app.setPalette(...)` без явной просьбы — это меняет вид всего приложения.
- Никогда не пиши `widget.setStyleSheet("background: ...")` без класс/ID-селектора, если внутри есть QLabel/QFrame без своего bg — каскадно покрасишь все детали.

## Архитектура: никакого хардкода бизнес-данных

Полное правило — в [CLAUDE.md](../../../CLAUDE.md#архитектурное-правило-никакого-хардкода-бизнес-данных). Кратко для UI-задач:

- **Списки/dropdown'ы/чипы причин/типов/тегов**, которые пользователь может захотеть редактировать — никогда не литералы в Python-коде; только `client.get('/<resource>/')` или `state.get_<resource>()`.
- Если для экрана нужен новый справочник, которого нет — сначала **создаёшь модель + миграцию + CRUD-эндпоинт + сидер** в backend, потом UI настройки в `pos/screens/settings_sections/`, **только потом** UI экрана его потребляет.
- В `pos/state.py` — lazy-load кэш + метод `invalidate_<resource>` (вызывается из соответствующей секции настроек после сохранения).
- В тестах виджета — мокаем `client.get('/<resource>/')`, не подсовываем хардкод-список из кода компонента.
- Дефолты для нового ресторана — в `apps/<bounded-context>/defaults.py`, оттуда сидим через миграцию + `post_save` сигнал на `Restaurant`.

Признак, что пора выносить в БД: **>2 вариантов И** их семантика — про бизнес ресторана (а не про state-машину ядра). Пример: «причины отмены» = бизнес → БД. «OrderStatus» = ядро → enum.

## Запреты

- **Нельзя** придумывать layout, цвета, паддинги, состояния — всё из `.pen`.
- **Нельзя** добавлять экраны/действия, которых нет в PRD. Если дизайн противоречит PRD — спросить пользователя, не молча выбирать одно.
- **Нельзя** реализовывать любой экран вне MVP-скоупа без явного подтверждения пользователем.
- **Нельзя** писать код waiter (PWA, `apps/waiter/`, `prd-v3/waiter/`) — это вне скоупа агента, см. [CLAUDE.md](../../../CLAUDE.md). Этот skill — только для PySide6-экранов кассира.
- **Нельзя** писать код owner-dashboard (`apps/dashboard/`, Next.js) — это вне скоупа агента.
- **Нельзя** использовать QML или Qt WebEngine — стек cashier'а это PySide6 QtWidgets + Python 3.12, и только.
- **Нельзя** пропускать Шаг 5 (сводку) даже если экран кажется тривиальным.
- **Нельзя** хардкодить бизнес-списки/строки в коде UI (причины отмены, скидки, шаблоны комментариев, способы оплаты сверх ядра, и т.д.) — только из БД через API. См. секцию «Архитектура: никакого хардкода бизнес-данных» выше.
